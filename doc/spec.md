## 1. 处理器概述

### 1.1 架构参数

| 参数       | 值                                    | 备注                                                              |
| ---------- | ------------------------------------- | ----------------------------------------------------------------- |
| 指令集架构 | RV32I                                 | 必做；RV32M 为加分项                                              |
| 流水线深度 | 5 级                                  | IF · ID · EX · MEM · WB                                           |
| 发射宽度   | 2（参数化 `N`）                       | `N` 固定为 2；是否支持 1/4 需在设计阶段明确决定，不留后续添加空间 |
| 执行单元   | 2× ALU                                | 数量随 `N` 自动生成                                               |
| 分支预测   | Bi-mode                               | 后期可迭代为 TAGE/RAS                                             |
| L1 D-Cache | 8 KB，4/8 路组相联                    | LRU 替换，写回+写分配                                             |
| L1 I-Cache | 独立，与 D-Cache 分离                 | 加分项                                                            |
| CSR        | `mcycle`、`mtime`（MMIO）、`minstret` | 必做                                                              |
| 预取器     | Next-line                             | CSR 控制开关；可迭代为 Stride/Stream                              |
| 测试基准   | Dhrystone（必做），Coremark（加分）   |                                                                   |
| HDL        | Chisel 3.5.6 / Scala 2.13             | 兼容 Java 8                                                       |

### 1.2 微架构总览

本处理器采用**顺序双发射五级流水线**架构。主线组实现顺序执行；突击组独立实现乱序执行变体，两者共享相同的外部内存总线协议与 CSR 接口规范，以保证最终集成兼容性。

流水线各级的主要职责如下：

- **IF（取指）**：基于 PC 从 I-Cache 获取最多 2 条对齐指令，同时查询分支预测器以推测下一 PC。
- **ID（译码）**：对两个指令槽并行译码，生成控制信号、立即数和寄存器地址；检测发射槽间的内部 RAW 冲突。
- **EX（执行）**：两个 ALU 并行计算；分支条件在本级判定，错误预测触发 Flush。
- **MEM（访存）**：访问 D-Cache；处理 Load 的符号/零扩展；Store 数据写入；PMA 地址检查。
- **WB（写回）**：从 ALU 结果、内存数据、PC+4、CSR 读取值中选择写回目标寄存器。

## 2. 顶层接口

### 2.1 顶层模块 `Top`

`Top` 模块是整个系统的最外层包装，包含核心（`Core`）、`IMem`（或 I-Cache）、`DMem`（或 D-Cache）三个主要子模块。

`Top` 模块对外暴露以下测试环境信号：

| 信号名            | 方向   | 位宽 | 描述                                                                                                         |
| ----------------- | ------ | ---- | ------------------------------------------------------------------------------------------------------------ |
| `status`          | Output | 1    | **核心架构标识信号**（编译期决定，运行期固定）：<br>0: "InOrderCore" (顺序双发射)<br>1: "OoOCore" (乱序执行) |
| `mem.req`         | Output | —    | 外部内存请求（Ready-Valid，见 §10 [模块间握手协议](#10-模块间握手协议)）                                     |
| `mem.resp`        | Input  | —    | 外部内存响应（Ready-Valid，见 §10 [模块间握手协议](#10-模块间握手协议)）                                     |
| `success`         | Output | 1    | 测试程序成功退出标志                                                                                         |
| `printChar.valid` | Output | 1    | 当前周期有字符输出                                                                                           |
| `printChar.bits`  | Output | 8    | 输出字符的 ASCII 值                                                                                          |
| `debugPc`         | Output | 32   | 当前 IF 级 PC，用于仿真调试                                                                                  |

为了保证两套架构的代码独立性，严禁在单一 Core 模块内部使用逻辑判断强行混合两套架构。必须分别实现 InOrderCore 和 OoOCore 两个独立模块，并根据顶层参数选择性例化。

### 2.2 外部内存总线 `MemBusIO`  

所有片外访问（I-Cache Miss、D-Cache Miss）均经由统一的内存总线。

**请求通道（`req`，CPU → MEM）：**

| 字段    | 位宽 | 描述                       |
| ------- | ---- | -------------------------- |
| `addr`  | 32   | 访问地址，字节寻址         |
| `wdata` | 32   | 写数据（读请求时忽略）     |
| `wen`   | 1    | 写使能，高有效             |
| `wmask` | 4    | 字节写使能（Store 时有效） |

**响应通道（`resp`，MEM → CPU）：**

| 字段    | 位宽 | 描述                       |
| ------- | ---- | -------------------------- |
| `rdata` | 32   | 读返回数据（写请求时忽略） |

内存总线延迟固定为 **10 个时钟周期**（自 `req` 握手起至 `resp.valid` 有效止）。

## 3. 流水线前端

### 3.1 PC 生成逻辑

PC 在每个时钟上升沿更新。更新优先级从高到低为：

| 优先级    | 条件                          | 下一 PC 来源            | 惩罚说明         |
| --------- | ----------------------------- | ----------------------- | ---------------- |
| 1（最高） | 分支预测错误（EX 级反馈）     | EX 级计算的真实目标地址 | 2 周期 Flush     |
| 2         | **JAL 跳转指令（ID 级判定）** | `PC + imm`，低位清零    | **1 周期 Flush** |
| 3         | JALR 指令（EX 级计算）        | `rs1 + imm`，低位清零   | 2 周期 Flush     |
| 4         | 流水线 Stall                  | 保持当前 PC 不变        | -                |
| 5         | 分支预测器预测跳转            | BPU 给出的预测目标      | 0 周期           |
| 6（最低） | 默认顺序取指                  | `PC + N * 4`            | -                |


**注： JAL 的 redirect 在 ID 级当拍生效，并覆盖 IF 的预测结果。否则会为2周期**。双发射时，IF 级每周期向 ID 级送出最多 **2 条** 32 位指令。若 PC 不对齐或 I-Cache 返回不足，槽 1 的 `valid` 置为 0，本周期退化为单发射。

#### 3.1.1 同周期多源 Redirect 仲裁规则

同一周期只允许一个 redirect 来源生效，优先级：`EX（分支误预测 / JALR）> ID（JAL）> IF（BPU）`。PC MUX 选择信号须互斥编码（one-hot 或 priority encoder）；JALR 与分支误预测共享 `exRedirect.valid`，不会在同一周期并发。

#### 3.1.2 Flush 在 N 路发射下的作用粒度

Flush 以整拍为粒度：flush 有效时，目标流水线寄存器内**所有 N 个槽位**全部清零（NOP bubble），不存在"保留 slot0 仅清 slot1"的部分 flush。`kill` 是 flush 在槽粒度的体现，整拍 flush 时所有槽的 `kill` 同时置 1；`kill` 不可单独用于部分 flush。

**Flush 优先级高于 Stall**，flush 有效时无论 stall 状态如何均写入 bubble：`when(flush) { reg := bubble } .elsewhen(!stall) { reg := next }`。

> 取指对齐导致的单槽 `valid=0` 属于正常 `valid` 语义，不属于 flush（见 §4.1.1）。

### 3.2 I-Cache 接口  

I-Cache 对 Core 暴露如下接口：

| 信号名        | 方向   | 位宽 | 描述                             |
| ------------- | ------ | ---- | -------------------------------- |
| `req.addr`    | Output | 32   | 取指基地址（以 N*4 字节对齐）    |
| `req.valid`   | Output | 1    | 本周期取指请求有效               |
| `resp.insts`  | Input  | 32*N | 包含 N 个指令槽的向量 (`Vec`)    |
| `resp.valids` | Input  | N    | 对应 N 个指令槽的有效位 (`Vec`)  |
| `resp.valid`  | Input  | 1    | Cache 命中，当前 Bundle 整体有效 |  |

I-Cache 未命中时，`resp.valid` 保持低电平，流水线 Stall 直至数据从内存返回（10 周期）。

### 3.3 分支预测单元（BPU）

本处理器实现 **Bi-mode 分支预测器**，由三张表构成：

| 结构            | 条目数  | 索引方式 | 位宽/项        | 作用               |
| --------------- | ------- | -------- | -------------- | ------------------ |
| Choice Table    | 512     | PC[10:2] | 2-bit 饱和计数 | 决定参考哪张方向表 |
| Taken Table     | 512     | PC[10:2] | 2-bit 饱和计数 | 预测"跳转"         |
| Not-Taken Table | 512     | PC[10:2] | 2-bit 饱和计数 | 预测"不跳转"       |
| BTB             | 256     | PC[9:2]  | 32-bit         | 存储分支目标地址   |
| RAS（加分项）   | 16 深度 | 栈       | 32-bit         | 函数返回地址预测   |

**预测逻辑：** Choice Table 的当前值决定读取 Taken Table 还是 Not-Taken Table；选中表的高位（bit[1]）决定最终预测结果。

**BPU 接口  ：**

| 信号名         | 方向（对 IF） | 位宽 | 描述            |
| -------------- | ------------- | ---- | --------------- |
| `queryPc`      | Input         | 32   | 查询 PC         |
| `predTaken`    | Output        | 1    | 预测是否跳转    |
| `predTarget`   | Output        | 32   | 预测目标地址    |
| `updateValid`  | Input         | 1    | EX 级更新使能   |
| `updatePc`     | Input         | 32   | 被更新的分支 PC |
| `updateTaken`  | Input         | 1    | 实际跳转结果    |
| `updateTarget` | Input         | 32   | 实际目标地址    |

BPU 更新在 EX 级末尾写入，下一周期生效。预测错误时，EX 级向 IF 发出 Flush 信号，清空 IF/ID 寄存器，并以真实目标地址重新取指（2 周期惩罚）。

**竞态处理（Bypass）：** BPU 内部 SRAM 必须实现 Write-First（写优先）逻辑。若同一周期 IF 级查询的 PC 与 EX 级更新的 PC 命中，查询接口应直接前递（Bypass）刚写入的新预测状态。

### 3.4 高级分支预测（加分项）

本节描述 **TAGE 预测器** 与 **RAS** 的设计规格，作为 §3.3 Bi-mode 预测器的迭代替换方案。两者均通过与 §3.3 完全相同的 BPU 接口与流水线交互，不改变接口定义；仅内部预测逻辑升级。

#### 3.4.1 TAGE 预测器

TAGE（TAgged GEometric length predictor）利用多张以不同历史长度索引的带标签表，捕捉更长的指令相关性。本设计采用 **1 张基础表（T0）+ 4 张带标签表（T1–T4）** 的简化 TAGE。

**表结构：**

| 表           | 条目数 | 索引方式                        | 每项字段                                                     | 历史长度 |
| ------------ | ------ | ------------------------------- | ------------------------------------------------------------ | -------- |
| T0（基础表） | 1024   | `PC[11:2]`                      | `ctr`（2-bit 饱和计数）                                      | —        |
| T1           | 256    | `PC[9:2] XOR GHR[7:0]`          | `tag`（8-bit）、`ctr`（3-bit 饱和计数）、`u`（2-bit 有用位） | 8        |
| T2           | 256    | `PC[9:2] XOR GHR[15:0]`（折叠） | 同 T1                                                        | 16       |
| T3           | 256    | `PC[9:2] XOR GHR[31:0]`（折叠） | 同 T1                                                        | 32       |
| T4           | 256    | `PC[9:2] XOR GHR[63:0]`（折叠） | 同 T1                                                        | 64       |

全局历史寄存器（GHR）宽度为 **64 位**，每周期移位追加实际跳转结果。带标签表的索引中超出 GHR 位数的部分通过折叠异或（fold-XOR）压缩到对应位宽。

**预测逻辑：**
1. 同时查询 T0–T4；T1–T4 中命中（`tag` 匹配）且历史长度最长的表为**提供者（provider）**；次长命中表为**备用提供者（altpred）**。
2. 若提供者命中，以其 `ctr` 高位作为预测结果；否则回退至 T0（基础表）。
3. `u` 位为 0 且提供者预测错误时，备用预测者结果可替代输出（Usefulness-based fallback）。

**更新逻辑（EX 级末尾）：**
- 更新提供者的 `ctr`（饱和计数）。
- 若预测错误，在未命中的带标签表中分配一个新项（从最短历史表开始，找 `u=0` 的项覆盖）；新项 `ctr` 初始化为弱正确（`3'b100`）。
- 若提供者预测正确且结果与备用提供者不同，`u` 位加一；若提供者预测错误且备用预测者正确，`u` 位减一。
- 每 256K 周期对所有带标签表的 `u` 位做一次全局右移（Periodic Reset），避免有用位长期饱和。

**时序约束：** TAGE 查询须在 **1 个流水周期内完成**（组合逻辑或寄存器输出均可，但若采用寄存器输出则需提前一拍发起查询）；更新写入可在 EX 级末尾或下一拍延迟写入，写入前须先完成读-修改-写。

#### 3.4.2 返回地址栈（RAS）

RAS 用于加速函数返回（`JALR` 且 `rd=x0, rs1=ra`）的目标地址预测，完全消除此类 JALR 的误预测惩罚。

**规格：**

| 参数      | 值                                                                                                                                                    |
| --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| 深度      | 16 项（循环栈，溢出时最旧项被覆盖）                                                                                                                   |
| 栈指针    | 4 位计数器，`push` 时加一，`pop` 时减一                                                                                                               |
| 每项字段  | `retAddr`（32-bit，`PC + 4`）                                                                                                                         |
| Push 时机 | ID 级识别 `JAL/JALR rd=ra` 时，将 `PC + 4` 压栈                                                                                                       |
| Pop 时机  | ID 级识别 `JALR rd=x0, rs1=ra` 时，弹出栈顶作为预测目标地址                                                                                           |
| 错误恢复  | EX 级确认 JALR 真实目标后，若与 RAS 预测不符，触发 2 周期 Flush 并恢复栈指针至 Pop 前状态（通过与 IF/ID 流水线寄存器同步的 `rasCheckpoint` 快照实现） |

**RAS 接口（融入 §3.3 BPU 接口）：**

| 信号名           | 方向（对 IF/ID） | 位宽 | 描述                               |
| ---------------- | ---------------- | ---- | ---------------------------------- |
| `ras.push`       | Input            | 1    | ID 级请求压栈（JAL/JALR 调用）     |
| `ras.pushAddr`   | Input            | 32   | 压栈返回地址（`PC + 4`）           |
| `ras.pop`        | Input            | 1    | ID 级请求弹栈（函数返回 JALR）     |
| `ras.topAddr`    | Output           | 32   | 栈顶地址，作为 JALR 的预测目标地址 |
| `ras.flush`      | Input            | 1    | EX 级触发，恢复栈指针至 checkpoint |
| `ras.checkpoint` | Input            | 4    | EX 级提供的正确栈指针快照          |

## 4. 译码与发射

### 4.1 双路并行译码

ID 级包含 **2 个独立译码器**，分别处理 IF/ID 寄存器的槽 0 和槽 1。每个译码器输出一个 `DecodedSlot`，内含完整的控制信号集合。

**译码器输出信号 `DecodedSlot`  ：**

| 字段                  | 位宽           | 描述                                       |
| --------------------- | -------------- | ------------------------------------------ |
| `pc`                  | 32             | 本条指令的 PC                              |
| `rs1Addr` / `rs2Addr` | 5              | 源寄存器地址                               |
| `rdAddr`              | 5              | 目标寄存器地址                             |
| `rs1Data` / `rs2Data` | 32             | 从寄存器堆读出的源数据                     |
| `imm`                 | 32（符号扩展） | 立即数                                     |
| `aluOp`               | 5              | ALU 操作码（共 12 种）                     |
| `op1Sel`              | 2              | ALU 操作数 1 来源：RS1 / PC / 0            |
| `op2Sel`              | 2              | ALU 操作数 2 来源：RS2 / IMM / PC+4        |
| `wbSel`               | 2              | 写回数据来源：ALU / MEM / PC+4 / CSR       |
| `rfWen`               | 1              | 寄存器写使能                               |
| `memRen`              | 1              | Load 操作使能                              |
| `memWen`              | 1              | Store 操作使能                             |
| `memWd`               | 2              | 访存宽度：字（00）/ 半字（01）/ 字节（10） |
| `memSigned`           | 1              | Load 数据是否符号扩展                      |
| `brType`              | 3              | 分支类型（0=无，1=BEQ，2=BNE，…）          |
| `isJump`              | 1              | 是否为 JAL/JALR 跳转指令                   |
| `valid`               | 1              | 本槽指令是否有效                           |

#### 4.2 流水线寄存器控制位语义规范

为保证 flush 时数据不残留、bubble 正确传播，**每一级流水线寄存器必须维护以下三类控制位**，语义统一如下：

| 控制位    | 位宽 | 语义                                                                                 |
| --------- | ---- | ------------------------------------------------------------------------------------ |
| `valid`   | 1    | 本槽当前持有有效指令（`0` = NOP bubble，所有写使能/访存使能必须同时被 `valid` 门控） |
| `kill`    | 1    | 本周期本槽被上游 flush 作废；接收方应将 `valid` 强制清零，不执行任何副作用操作       |
| `allowIn` | 1    | 本级准备好接收新数据（对应 Ready-Valid 握手中的 `ready`）；`allowIn=0` 时上游须冻结  |

实现约定：
- `kill=1` 时写入全零（NOP bubble），**flush 优先于 stall**（见 §3.1.3）。
- `valid=0` 的槽位不得触发写回、访存、CSR 等有副作用的动作。
- 双发射中槽 0 和槽 1 各自独立持有上述三个控制位，不共享。
- `kill` 是整拍 flush 在槽粒度的体现，不可单独用于部分 flush（见 §3.1.3）。

ID 级需检测**槽间内部 RAW 冲突**：若槽 0 的 `rdAddr` 与槽 1 的 `rs1Addr` 或 `rs2Addr` 相同，且槽 0 写寄存器（`rfWen=1`），则槽 1 必须等待槽 0 完成写回。处理方式为**本周期降级单发射**：仅发射槽 0，槽 1 在下一周期重新尝试发射。

#### 4.3 发射选择语义：前缀连续发射（Prefix Issue）

采用**前缀连续发射**策略：某槽无法发射时，其后所有槽全部变为 bubble，不允许跳过发射后续指令。各槽停发条件：

| 条件                                     | 影响槽位                |
| ---------------------------------------- | ----------------------- |
| 槽 0 `valid=0`（取指包不足/对齐问题）    | 槽 0 及槽 1 均为 bubble |
| 槽 1 `valid=0`（取指包仅返回 1 条）      | 仅槽 1 为 bubble        |
| 槽 1 与槽 0 存在 RAW 冲突（见 §4.2）     | 仅槽 1 为 bubble        |
| Load-Use stall（HazardUnit 触发）        | 槽 0 和槽 1 均冻结      |
| D-Cache / I-Cache Miss（全流水线 stall） | 所有槽冻结              |

### 4.4 立即数生成规则

| 指令类型 | 立即数字段                                             | 符号扩展 |
| -------- | ------------------------------------------------------ | -------- |
| I 型     | `inst[31:20]`                                          | 是       |
| S 型     | `{inst[31:25], inst[11:7]}`                            | 是       |
| B 型     | `{inst[31], inst[7], inst[30:25], inst[11:8], 1'b0}`   | 是       |
| U 型     | `{inst[31:12], 12'b0}`                                 | 是       |
| J 型     | `{inst[31], inst[19:12], inst[20], inst[30:21], 1'b0}` | 是       |

## 5. 执行级

### 5.1 ALU 操作码定义

ALU 支持以下操作，操作码宽度为 5 位：

| 操作码  | 助记符      | 运算                           |
| ------- | ----------- | ------------------------------ |
| `5'd0`  | `ALU_ADD`   | `op1 + op2`                    |
| `5'd1`  | `ALU_SUB`   | `op1 - op2`                    |
| `5'd2`  | `ALU_AND`   | `op1 & op2`                    |
| `5'd3`  | `ALU_OR`    | `op1 \| op2`                   |
| `5'd4`  | `ALU_XOR`   | `op1 ^ op2`                    |
| `5'd5`  | `ALU_SLL`   | `op1 << op2[4:0]`              |
| `5'd6`  | `ALU_SRL`   | `op1 >> op2[4:0]`（逻辑）      |
| `5'd7`  | `ALU_SRA`   | `op1 >>> op2[4:0]`（算术）     |
| `5'd8`  | `ALU_SLT`   | `(op1 signed < op2) ? 1 : 0`   |
| `5'd9`  | `ALU_SLTU`  | `(op1 unsigned < op2) ? 1 : 0` |
| `5'd10` | `ALU_LUI`   | `op2`（直接传递）              |
| `5'd11` | `ALU_COPY1` | `op1`（用于 JALR 地址计算）    |

### 5.2 操作数选择

| `op1Sel`            | 来源                 | 典型指令   |
| ------------------- | -------------------- | ---------- |
| `2'd0`（`OP1_RS1`） | 寄存器 rs1（经旁路） | 大多数指令 |
| `2'd1`（`OP1_PC`）  | 当前指令 PC          | AUIPC、JAL |
| `2'd2`（`OP1_IMM`） | 立即数（零值）       | LUI        |

| `op2Sel`            | 来源                 | 典型指令                 |
| ------------------- | -------------------- | ------------------------ |
| `2'd0`（`OP2_RS2`） | 寄存器 rs2（经旁路） | R 型指令                 |
| `2'd1`（`OP2_IMM`） | 立即数               | I/S/B/U/J 型指令         |
| `2'd2`（`OP2_PC4`） | `PC + 4`             | JAL/JALR（写回返回地址） |

### 5.3 旁路前递网络  

双发射下旁路网络的源优先级（高→低）：

| 优先级      | 旁路来源                            | 说明            |
| ----------- | ----------------------------------- | --------------- |
| 1           | EX/MEM 寄存器（本周期 ALU 结果）    | 覆盖 EX-EX RAW  |
| 2           | MEM/WB 寄存器（ALU 结果或访存结果） | 覆盖 MEM-EX RAW |
| 3           | WB 级写回数据                       | 覆盖 WB-EX RAW  |
| 0（不旁路） | 寄存器堆读出值                      | 无冒险时使用    |

每个 EX 级 ALU 的两个操作数各自独立检查上表，共需 4 套旁路 MUX（2 个槽 × 2 个操作数）。旁路时需同时比较来源的 `rdAddr`，且 `rdAddr ≠ 0`（x0 恒为零，不旁路）。

#### 5.3.1 完整 Forwarding 路径矩阵

下表列出双发射下**所有必须实现的旁路路径**，其中"源"指产生结果的槽/级，"目的"指消费结果的槽/级：

| 路径编号 | 源                         | 目的                   | 条件                                         | 说明                                 |
| -------- | -------------------------- | ---------------------- | -------------------------------------------- | ------------------------------------ |
| F1       | slot0.EX/MEM（本周期结果） | slot0.EX（下周期）     | `slot0_ex.rd == slot0_next.rs{1,2}`          | EX→EX，跨周期同槽                    |
| F2       | slot0.EX/MEM（本周期结果） | slot1.EX（下周期）     | `slot0_ex.rd == slot1_next.rs{1,2}`          | EX→EX，跨周期跨槽                    |
| F3       | slot1.EX/MEM（本周期结果） | slot0.EX（下周期）     | `slot1_ex.rd == slot0_next.rs{1,2}`          | EX→EX，跨周期跨槽                    |
| F4       | slot1.EX/MEM（本周期结果） | slot1.EX（下周期）     | `slot1_ex.rd == slot1_next.rs{1,2}`          | EX→EX，跨周期同槽                    |
| **F5**   | **slot0.EX（同周期）**     | **slot1.EX（同周期）** | `slot0.rd == slot1.rs{1,2}`（且无 Load-Use） | **双发射同周期内槽间旁路，必须实现** |
| F6       | slot0.MEM/WB               | slot0.EX（下周期）     | `slot0_mem.rd == slot0_next.rs{1,2}`         | MEM→EX，同槽                         |
| F7       | slot0.MEM/WB               | slot1.EX（下周期）     | `slot0_mem.rd == slot1_next.rs{1,2}`         | MEM→EX，跨槽                         |
| F8       | slot1.MEM/WB               | slot0.EX（下周期）     | `slot1_mem.rd == slot0_next.rs{1,2}`         | MEM→EX，跨槽                         |
| F9       | slot1.MEM/WB               | slot1.EX（下周期）     | `slot1_mem.rd == slot1_next.rs{1,2}`         | MEM→EX，同槽                         |
| F10      | slot0.WB                   | slot{0,1}.EX           | `slot0_wb.rd == slot{0,1}.rs{1,2}`           | WB→EX                                |
| F11      | slot1.WB                   | slot{0,1}.EX           | `slot1_wb.rd == slot{0,1}.rs{1,2}`           | WB→EX                                |

> **注意**：F5（slot0.EX → slot1.EX 同周期旁路）仅在 slot0 产生的不是 Load 结果时成立；若 slot0 为 Load，则 slot1 依赖 slot0 的情形必须在 ID 级检测并降级为单发射（见 §4.2 和 Load-Use 停顿规则）。

#### 5.3.2 Forwarding 判定与 Stall 决策顺序

**先判断 forwarding 是否能解决冲突，再决定是否 stall**（顺序写反会产生不必要的 stall）。判定步骤：

1. 对每条指令的每个源寄存器，遍历 F1–F11；若命中（源 `valid=1`，`rdAddr` 匹配且 `≠ 0`），标记可旁路，不产生 stall。
2. **例外**：Load-Use（源为 Load 且在 EX 级）——无论是否有旁路路径，均必须 stall 1 周期。
3. 仅当步骤 1 无旁路且不属于步骤 2 时，才触发 stall。

附加约束：F1–F11 全部实现（含跨槽跨周期路径）；CSR 读取结果同等接入旁路网络。

### 5.4 分支与跳转处理

| 指令类型                  | 目标地址计算                                       | EX 级惩罚                                                |
| ------------------------- | -------------------------------------------------- | -------------------------------------------------------- |
| BEQ/BNE/BLT/BGE/BLTU/BGEU | `PC + imm`（B 型立即数）                           | 预测错误时 2 周期 Flush                                  |
| JAL                       | `PC + imm`（J 型立即数），**ID 级计算并 redirect** | **1 周期 Flush（ID 级），EX 级仅写 rd=PC+4，不再 flush** |
| JALR                      | `(rs1 + imm) & ~1`                                 | 2 周期 Flush                                             |

分支条件判断在 EX 级基于经旁路的 `rs1`/`rs2` 进行，结果通过 `brTaken` 信号回馈 HazardUnit 和 BPU 更新端口。JAL 的 redirect 在 ID 级完成（见 §3.1 实现强制要求），EX 级不对 JAL 产生第二次 PC 修改。

### 5.5 RV32M 乘除法单元（加分项）

RV32M 乘除法作为 EX 级独立的**多周期执行单元**插入，通过 Ready-Valid 握手与流水线交互。乘法典型延迟为 3 周期，除法典型延迟为 32 周期（可采用 Chisel 内建算子实现后续替换为 Blackbox）。

## 6. 访存级与存储子系统

### 6.1 D-Cache 规格  

| 参数            | 值                                           | 备注                                    |
| --------------- | -------------------------------------------- | --------------------------------------- |
| 总容量          | 8 KB                                         | 参数化：可配置为 4/8/16/32 KB（加分项） |
| Cache Line 大小 | 64 B（16 × 32-bit 字）                       | 固定                                    |
| 组相联度        | 4 路（默认）或 8 路                          | 参数 `nWays`                            |
| 组数（4 路时）  | 32 组                                        | `8192 / (64 × 4) = 32`                  |
| 索引位          | `addr[10:6]`（5 位）                         |                                         |
| 块内偏移位      | `addr[5:0]`（6 位）                          |                                         |
| 标签位          | `addr[31:11]`（21 位）                       |                                         |
| 替换策略        | LRU（或伪 LRU）                              | **必做**                                |
| 写策略          | 写回（Write-back）+ 写分配（Write-allocate） |                                         |
| 命中延迟        | 1 周期                                       |                                         |
| 缺失延迟        | 10 周期（全流水线 Stall）                    |                                         |

### 6.2 D-Cache 接口  

**CPU 侧请求（`req`，来自 MEM 级，Ready-Valid）：**

| 字段     | 位宽 | 描述                     |
| -------- | ---- | ------------------------ |
| `flush`  | 1    | 复位清理 Cache 信号      |
| `addr`   | 32   | 访问地址                 |
| `wdata`  | 32   | Store 数据               |
| `wen`    | 1    | 写使能（Store）          |
| `memRen` | 1    | 读使能（Load）          |
| `wmask`  | 4    | 字节使能（SB/SH/SW）     |
| `memWd`  | 2    | 访存宽度（字/半字/字节） |
| `signed` | 1    | Load 是否符号扩展        |

**CPU 侧响应（`resp`，返回给 MEM 级，Ready-Valid）：**

| 字段    | 位宽 | 描述                          |
| ----------| ---- | ----------------------------- |
| `rdata`   | 32   | Load 返回数据（Store 时忽略） |
| `missOut` | 32   | Cache 未命中信号，高电平期间流水线应 stall |

### 6.3 D-Cache Miss 处理

D-Cache 缺失时，**全流水线 Stall**（含 IF、ID、EX、MEM、WB 各级），直至内存总线返回数据（固定 10 周期）并完成 Cache Line 回填后，流水线才恢复推进。本设计**不支持 Inflight 访存**，即 D-Cache Miss 期间不允许新的访存请求发出。

> **注**：不实现 Miss Buffer 和 Store-to-Load Forwarding。Store Buffer 的 drain 操作在 Miss 期间同样暂停。

#### 双发射 Load/Store 同周期访存仲裁

D-Cache 提供 **1 个读写端口**（单端口，同周期只能处理 1 笔访存请求）。双发射情况下同周期最多出现 1 条 Load 和 1 条 Store，仲裁规则如下：

| 场景                  | 处理方式                                                                                  |
| --------------------- | ----------------------------------------------------------------------------------------- |
| 槽 0 Load，槽 1 Store | 槽 0 Load 优先访问 D-Cache；槽 1 Store 延迟至下一周期（触发 1 周期 stall，槽 1 重新发射） |
| 槽 0 Store，槽 1 Load | 槽 0 Store 优先；槽 1 Load 延迟（同上）                                                   |
| 两槽均为 Load         | 不允许双发射同时为 Load（ID 级冲突检测降级为单发射，见 §4.2）                             |
| 两槽均为 Store        | 不允许双发射同时为 Store（同上降级）                                                      |
| 仅一槽访存            | 正常流水，无仲裁开销                                                                      |

**Store 处理**：顺序核中 Store 在 MEM 级直接写入 D-Cache（或等待 D-Cache Miss 解决后写入），不使用 Store Buffer，不进行 Store-to-Load Forwarding。

### 6.4 Load 数据符号/零扩展

| `memWd`        | `signed`    | 返回数据处理                   |
| -------------- | ----------- | ------------------------------ |
| 字（`2'd0`）   | 任意        | 直接返回 32 位原始数据         |
| 半字（`2'd1`） | 1（有符号） | `{{16{data[15]}}, data[15:0]}` |
| 半字（`2'd1`） | 0（无符号） | `{16'b0, data[15:0]}`          |
| 字节（`2'd2`） | 1（有符号） | `{{24{data[7]}}, data[7:0]}`   |
| 字节（`2'd2`） | 0（无符号） | `{24'b0, data[7:0]}`           |

字节/半字的实际位置由地址低位 `addr[1:0]` 确定，MEM 级从返回的 32 位字中提取对应字节/半字。

### 6.5 PMA 与 printf 绕过

物理内存属性（PMA）检查在 MEM 级进行。本设计需处理以下两类特殊地址，均属 **Uncacheable** 区域，访问不经过 D-Cache：

**printf 输出寄存器（`0x10001FF1`）：** Store Byte 操作此地址时：

- `printChar.valid` 置高，`printChar.bits` 输出写数据低 8 位
- 该地址的 Load 行为未定义，不予处理

**`mtime` 实时计数器（`0xBFF8`–`0xBFFC`）：** 内存映射只读寄存器，Load Word 访问此地址范围时：

- 地址 `0xBFF8`：返回 `mtime` 计数器低 32 位（与 `mcycle` 同源，或由独立硬件计数器驱动）
- 地址 `0xBFFC`：返回 `mtime` 计数器高 32 位（32 位实现中可固定返回 `0`）
- Store 写入此地址范围无效，硬件忽略，不修改计数器值
- 两地址均属 Uncacheable，Load 须旁路 D-Cache 直接由 PMA 检查逻辑在 MEM 级截获并返回当前计数值

### 6.6 Next-line 预取器

预取器在每次 I-Cache **命中**后，自动向 I-Cache 预取下一个 Cache Line（`PC + 64`）。若目标 Cache Line 已在 I-Cache 中，本次预取请求静默丢弃；若已有一条预取请求在途，新请求同样丢弃，避免阻塞正常缺失。

通过 CSR `prefetch_ctrl`（自定义地址，见 §9.1）的 `bit[0]` 控制预取开关（1 = 开启，0 = 关闭），以便实验对比有/无预取的性能差异。处理器复位后 `prefetch_ctrl` 初始值由测试程序写入，默认不做假设。

**预取器接口（对内存总线）：**

| 信号名               | 方向（对内存总线） | 位宽 | 描述                                       |
| -------------------- | ------------------ | ---- | ------------------------------------------ |
| `pf.req.valid`       | Output             | 1    | 预取请求有效                               |
| `pf.req.ready`       | Input              | 1    | 内存总线可接收预取请求                     |
| `pf.req.bits.addr`   | Output             | 32   | 预取目标 Cache Line 基地址（64B 对齐）     |
| `pf.req.bits.isPref` | Output             | 1    | 恒为 `1`，标识本请求来自预取器而非正常缺失 |

预取请求优先级**低于**正常缺失请求；总线忙时 `pf.req.ready` 置低，预取器本次请求丢弃，不重试。

### 6.7 Stride 预取器（加分项）

Stride 预取器通过追踪每个 Load 指令的访存步长，对规律性数据访问（如数组遍历）进行提前预取，可进一步降低 D-Cache 缺失惩罚。

**Reference Prediction Table（RPT）规格：**

| 参数         | 值                                                                                    |
| ------------ | ------------------------------------------------------------------------------------- |
| 表项数       | 64                                                                                    |
| 索引方式     | Load 指令 PC 的低 6 位（`PC[7:2]`）                                                   |
| 每项字段     | `tag`（PC 高位）、`prevAddr`（上次访存地址）、`stride`（步长）、`state`（4 状态 FSM） |
| 步长精度     | 字节粒度，有符号 32 位                                                                |
| 预取触发条件 | 状态机达到 `Steady` 或 `Transient` 稳定态                                             |
| 预取目标地址 | `prevAddr + stride`（按 Cache Line 对齐后发送给内存总线）                             |

**状态机（4 状态 FSM）：**

| 状态        | 含义               | 迁移规则                                  |
| ----------- | ------------------ | ----------------------------------------- |
| `Initial`   | 初始，步长未知     | 记录首次访问地址，进入 `Transient`        |
| `Transient` | 观测到一次步长     | 步长一致 → `Steady`；步长变化 → `No-pred` |
| `Steady`    | 步长稳定，触发预取 | 步长一致 → 保持；步长变化 → `No-pred`     |
| `No-pred`   | 步长不规律，不预取 | 记录新步长 → `Transient`                  |

**Stride 预取器对内存总线的接口与 §6.6 `pf.*` 信号同构**，`isPref` 同样置 `1`，优先级相同（均低于正常缺失请求）。

`prefetch_ctrl` CSR 的 `bit[1]`（1 = 开启 Stride 预取，0 = 关闭）独立控制 Stride 预取器，与 Next-line 预取器开关互不影响。

## 7. 写回级与寄存器堆

### 7.1 写回数据选择

| `wbSel`            | 写回数据来源     | 典型指令          |
| ------------------ | ---------------- | ----------------- |
| `2'd0`（`WB_ALU`） | EX 级 ALU 输出   | 算术/逻辑指令     |
| `2'd1`（`WB_MEM`） | MEM 级 Load 数据 | LW/LH/LB/LHU/LBU  |
| `2'd2`（`WB_PC4`） | `PC + 4`         | JAL/JALR          |
| `2'd3`（`WB_CSR`） | CSR 读取值       | CSRRW/CSRRS/CSRRC |

### 7.2 寄存器堆规格  

双发射架构要求 **4 读 2 写**的寄存器堆。

| 参数       | 值                                     |
| ---------- | -------------------------------------- |
| 寄存器数   | 32（x0–x31）                           |
| 读端口数   | 4（每个发射槽各 2 个）                 |
| 写端口数   | 2（每个发射槽各 1 个）                 |
| x0         | 固定为 0，忽略写操作                   |
| 写后读转发 | 支持（同周期写读同一地址时，读取新值） |

#### 7.2.1 双写端口冲突仲裁规则

两槽同周期写同一 `rd`（`rfWen=1` 且 `rdAddr ≠ 0`）时，**槽 1（较年轻）覆盖槽 0**。Port-1 写使能优先级高于 Port-0；不触发 Stall。

#### 7.2.2 CSR 写与 GPR 写的冲突模型

- CSR 写（`CSR[addr] <= wdata`）在 **EX 级末尾**完成，不占用 WB 级 GPR 写端口。
- CSR 读结果写回 `rd` 在 **WB 级**，与普通 ALU 写回竞争同一写端口，冲突时适用 §7.2.1 的槽优先级。
- CSR 读写数据相关须纳入旁路网络（见 §5.3 CSR 旁路约束）。

### 8.1 HazardUnit 输入/输出信号  

**输入信号：**

| 信号名        | 位宽              | 来源         | 描述                       |
| ------------- | ----------------- | ------------ | -------------------------- |
| `exMemRen`    | `Vec(2, Bool)`    | ID/EX 寄存器 | EX 级两槽是否为 Load       |
| `exRdAddr`    | `Vec(2, UInt(5))` | ID/EX 寄存器 | EX 级两槽目标寄存器地址    |
| `idRs1Addr`   | `Vec(2, UInt(5))` | IF/ID 寄存器 | ID 级两槽 rs1 地址         |
| `idRs2Addr`   | `Vec(2, UInt(5))` | IF/ID 寄存器 | ID 级两槽 rs2 地址         |
| `brTaken`     | `Bool`            | EX 级        | 分支/跳转实际发生          |
| `dcacheStall` | `Bool`            | D-Cache      | 访存 Miss 导致的流水线停顿 |

**输出信号：**

| 信号名       | 位宽   | 描述                              |
| ------------ | ------ | --------------------------------- |
| `stallIF`    | `Bool` | 停顿 IF 级（PC 保持不变）         |
| `stallID`    | `Bool` | 停顿 ID 级（IF/ID 寄存器冻结）    |
| `flushEX`    | `Bool` | 向 EX 级插入气泡（load-use 解决） |
| `flushIF`    | `Bool` | 清空 IF/ID 寄存器（分支预测错误） |
| `flushID`    | `Bool` | 清空 ID/EX 寄存器（分支预测错误） |
| `slot1Stall` | `Bool` | 槽 1 因槽内 RAW 降级为单发射      |

### 8.2 冒险类型与处理策略

| 冒险类型       | 触发条件                            | 处理方式                    | 流水线代价 |
| -------------- | ----------------------------------- | --------------------------- | ---------- |
| Load-Use RAW   | EX 级为 Load，其 rd == ID 级任一 rs | Stall IF+ID，向 EX 插气泡   | 1 周期     |
| 双发射槽间 RAW | 槽 0 的 rd == 槽 1 的 rs            | 槽 1 降级，本周期仅发射槽 0 | 1 周期     |
| 分支预测错误   | EX 级 `brTaken` 与 BPU 预测不符     | Flush IF+ID，重新取指       | 2 周期     |
| JAL            | ID 级识别跳转目标，触发 redirect    | Flush IF                    | 1 周期     |
| JALR           | EX 级计算目标地址                   | Flush IF+ID                 | 2 周期     |
| D-Cache Miss   | D-Cache `resp.valid=0`              | 全流水线 Stall              | 10 周期    |
| I-Cache Miss   | I-Cache `resp.valid=0`              | Stall IF                    | 10 周期    |

### 8.3 复位（Reset）行为

处理器在 `reset` 信号有效（高电平，同步复位）期间执行以下初始化：

| 资源              | 复位后状态                                                                |
| ----------------- | ------------------------------------------------------------------------- |
| PC 寄存器         | 复位至 `0x80000000`（程序入口，与 QEMU/RISC-V 约定一致）                  |
| 所有流水线寄存器  | 清零（等效全插入 NOP bubble；所有 `valid` 位置 0）                        |
| 寄存器堆（GPR）   | 不强制清零（初始值未定义；程序须自行初始化所需寄存器）                    |
| CSR               | `mcycle = 0`，`minstret = 0`，`mstatus.MIE = 0`，其余按 RISC-V 规范初始值 |
| Store Buffer      | 清空（`valid` 全置 0）                                                    |
| D-Cache / I-Cache | 清空所有 valid 位（冷启动状态）                                           |
| BPU（分支预测器） | 所有 2-bit 计数器重置为弱不跳转（`2'b01`），BTB 全部无效                  |
| HazardUnit 状态   | 所有 stall/flush 信号清零                                                 |

`reset` 拉低后第一个时钟上升沿，PC 开始从 `0x80000000` 取指，流水线进入正常推测流。

### 8.4 非法指令处理

本课设**不要求**非法指令检测、精确异常、陷入（trap）及中断（interrupt）等机制。译码到未知编码的指令时，行为未定义，可静默丢弃或置 NOP，无需触发异常流程。

### 9.2 CSR 指令语义

## 10. 模块间握手协议

### 10.1 Ready-Valid 协议规范  

所有跨模块的数据流均采用 Chisel `Decoupled` 标准（即 Ready-Valid 协议）。协议约束如下：

| 约束          | 描述                                                             |
| ------------- | ---------------------------------------------------------------- |
| 握手成立      | `valid && ready` 同时为高的时钟上升沿                            |
| Producer 约束 | 一旦拉高 `valid`，在握手成立前**不得撤回** `valid` 或改变 `bits` |
| Consumer 约束 | 在 `valid=0` 时**不得**将自身内部状态视为数据已到达              |
| 背压传播      | 下游 `ready=0` 时，上游必须 Stall 或缓存数据                     |

### 10.2 内存总线时序

| 周期      | 信号状态                          | 说明                   |
| --------- | --------------------------------- | ---------------------- |
| T+0       | `req.valid=1, req.bits=(addr,…)`  | CPU 发出请求           |
| T+1       | `req.ready=1`                     | 内存接受请求，握手成立 |
| T+1 ~ T+9 | —                                 | 内存访问流水           |
| T+10      | `resp.valid=1, resp.bits.rdata=…` | 数据返回               |
| T+10      | `resp.ready=1`                    | CPU 消费数据，握手成立 |

### 10.3 流水线级间背压

对于可能产生多周期延迟的模块（D-Cache、I-Cache、乘除法单元），采用反压机制：下游未就绪（`valid=0` 或内部 busy）时，向上游流水线发出 Stall 信号，冻结所有上游级寄存器直至操作完成。

### 13.1 取指级（IF）与其对外端口

IF 负责在**推测取指流**下维护 PC、驱动 I-Cache、向 BPU 查询并接收预测结果，以及在**后端改向**时同步到权威 PC。对下，它将**对齐后的指令字与对应 PC** 经 `toId` 送给 ID；对上与 I-Cache、BPU 及后端反馈相连。表中 **`icache.*`** 为指令 Cache 的 Ready-Valid 请求/响应；**`bpu.*`** 中 `query`/`pred` 为取指侧查询与预测返回，`update` 为执行或提交侧对预测表的训练回写；**`redirect.*`** 来自乱序后端（或提交级），优先级高于推测流；**`toId.*`** 与 §3.1 多发射语义一致：`toId.bits` 为**单条取指槽**的载荷，**N** 路取指时在实现侧并列 **N** 套同构信号（线名可自加下标）；`ready` 可与 ID 及更下游背压聚合，避免前端 overrun。**`icache.resp`** 与 §3.2 对应时，可将 `inst0`/`inst1`/`inst1Valid` 视为本表**单槽抽象**在 **N=2** 下的展开。

| 端口                        | 方向   | 位宽 | 说明                                                                   |
| --------------------------- | ------ | ---- | ---------------------------------------------------------------------- |
| `icache.req.addr`           | output | 32   | 取指地址（建议 8B 对齐）                                               |
| `icache.req.valid`          | output | 1    | 取指请求有效                                                           |
| `icache.req.ready`          | input  | 1    | I-Cache 可接收请求（Ready-Valid）                                      |
| `icache.resp.inst`          | input  | 32   | 单槽指令字（多槽时共 **N** 条同构线，表内不逐槽枚举）                  |
| `icache.resp.valid`         | input  | 1    | 命中且本 bundle 数据有效                                               |
| `icache.resp.nextSlotValid` | input  | 1    | 非首槽是否有效（`N≥2` 时需要；单槽可接 0，语义同 §3.2 `inst1Valid`）   |
| `bpu.queryPc`               | output | 32   | BPU 查询 PC                                                            |
| `bpu.predTaken`             | input  | 1    | 预测是否跳转                                                           |
| `bpu.predTarget`            | input  | 32   | 预测目标地址                                                           |
| `bpu.updateValid`           | input  | 1    | BPU 表项更新使能                                                       |
| `bpu.updatePc`              | input  | 32   | 被更新分支的 PC                                                        |
| `bpu.updateTaken`           | input  | 1    | 实际跳转结果                                                           |
| `bpu.updateTarget`          | input  | 32   | 实际目标地址                                                           |
| `redirect.flush`            | input  | 1    | 后端权威冲刷（误预测、异常、中断等）                                   |
| `redirect.pc`               | input  | 32   | 改向目标 PC                                                            |
| `toId.valid`                | output | 1    | 送往 ID 的取指包有效                                                   |
| `toId.ready`                | input  | 1    | ID 可接收（可与后端背压聚合后再送入 IF）                               |
| `toId.bits.pc`              | output | 32   | 该取指槽对应 PC                                                        |
| `toId.bits.inst`            | output | 32   | 该取指槽指令字                                                         |
| `toId.bits.slotValid`       | output | 1    | 该槽在本 bundle 内是否有效（首槽通常与 `toId.valid` 一致；多槽各一套） |

**说明**：IF **不**产生寄存器数据值，也不解析数据相关；I-Cache Miss 时 `resp.valid` 为低，IF 应停顿或重试取指，不向 ID 推进无效包。`redirect.flush` 有效时，内部取指缓冲中与推测路径不一致的内容应丢弃，并在下一拍起从 `redirect.pc` 重新填充流水线。

### 13.2 译码级（ID）与其对外端口

与 IF 的接口：

`fromIf.bits` 为**单条取指槽**载荷；**N** 路译码时在实现侧并列 **N** 套同构信号，表内只列一套。

| 端口                    | 方向   | 位宽 | 说明                       |
| ----------------------- | ------ | ---- | -------------------------- |

### 13.3 乱序新增重命名级（Rename）

Rename 将 ID 输出的**架构寄存器编号**解析为**物理寄存器编号**，并为进入乱序后端的每条 uop 分配 **ROB 项**；若指令写回寄存器（`rfWen`），还需从**空闲列表**取一枚新的物理寄存器作为 `rd` 的**推测映射**。重命名侧维护**两张 RAT**（§13.3.1）与**空闲列表**；与 **ID** 的边界见 §13.3.3，与 **保留站 / 发射队列（RS）**、**ROB** 的边界见 §13.3.4、§13.3.5，每拍操作流程见 §13.3.6。

#### 13.3.1 寄存器映射表（RAT）

本设计采用**两张 RAT**，表深均为 **32** 项（对应 RV32I 架构寄存器索引），表项为**物理寄存器编号**（位宽 `⌈log2 P⌉`，`P` 为物理寄存器个数）。`x0` 不按表分配新物理号：读侧可按实现固定为**零**或专用物理寄存器；写 `rd=x0` 时不更新任何 RAT 项。

**推测 RAT（Speculative RAT）**  
反映**当前推测执行**下「架构寄存器 → 物理寄存器」的映射。本阶段**只读、只写此表**：根据 `rs1Addr` / `rs2Addr` **读**得 `pRs1` / `pRs2`；当本条 uop 写寄存器（`rfWen=1` 且 `rd≠x0`）时，将 **推测 RAT[`rd`]** 更新为**刚从空闲列表弹出**的新 `pRd`。发射、执行、访存均基于该推测映射链推进。

**架构 RAT（Architectural RAT）**  
反映**已按序提交**后的**确定的**映射，**仅在 ROB 提交**写回寄存器的指令时更新：将 **架构 RAT[`rd`]** 置为该指令最终可见的物理寄存器（通常即本条提交的 `pRd`；旧映射由 ROB 项与空闲列表回收逻辑配合释放）。**本阶段不修改架构 RAT**。

**回滚与恢复**  
发生**分支误预测、异常或 Flush** 需丢弃推测状态时，须将**推测 RAT 恢复为与架构 RAT 一致**（典型实现：**`specRAT ← archRAT`** 整表覆盖），使重新取指与 Rename 所见的别名关系与**最后已提交点**一致。

**Flush 时 Free List 回收**：仅覆盖 `specRAT ← archRAT` 是不够的——若不同步回收未提交的物理寄存器，Free List 将持续泄漏直至耗尽。正确做法是：Flush 触发时，遍历 ROB 中所有 `valid=1` 但尚未提交（`done=0` 或未到 head 的推测项）的项，将其已分配的 `pRd` **全部压回 Free List**。实现上可采用 **ROB walk** 方案：从 `tail-1` 向 `head` 倒序扫描，逐项回收 `pRd`；或在 ROB 项中额外保存 `oldPhysRd`，Flush 时将 spec RAT 中被覆盖的旧映射恢复（checkpoint 方案）。两者须在设计阶段选定其一，不允许仅做 specRAT 覆盖而跳过 Free List 回收。

#### 13.3.2 空闲列表（Free List）

**空闲列表**保存当前**未被推测链占用的**物理寄存器编号集合，存储结构由**FIFO**实现。**分配（Rename 每周期）**：对每条有效且 `rfWen=1`（且 `rd≠x0`）的 uop，从表顶弹出一枚物理号作为新的 `pRd`，并写入**推测 RAT**。**回收（提交级）**：当 ROB 按序提交某条写寄存器指令时，其**旧物理映射**（被本条覆盖前的 `pRd`）通常重新压入空闲列表，供后续指令复用；同时按 §13.3.1 更新**架构 RAT**；具体与 ROB 项中是否保存 **previous physical rd** 字段相关。

当**空闲表空**而无法为新的 `rd` 分配物理寄存器时，本级**不得**再接收来自 ID 的新 uop，即将 **`fromId.ready=0`**。

#### 13.3.3 本级与 ID 级间接口

Rename **从 ID 接收**与 §13.2 **`toRename` 完全同构**的载荷；线名 **`fromId.*`** 与 ID 侧 **`toRename.*`** 一一相连。**`fromId.bits` 为单槽**；**N** 路时在实现侧并列 **N** 套，表内只列一套。当 **ROB 满**、**空闲表不可用**或**下游分派不允许**时，Rename **拉低 `fromId.ready`**，使 `toRename` 握手不成立，IF/ID 停顿。

**分派原子性约束（Dispatch Atomicity）：**
Rename 向后端的分派操作必须是原子的。必须同时满足以下三个条件，才能拉高 `toRs.valid`：
1. ROB 队列未满 (`!rob.full`)
2. 保留站对应槽位未满 (`!rs.full`)
3. 空闲列表有足够的物理寄存器 (`!freelist.empty`)
绝不允许部分组件写入成功而另一部分阻塞，否则会破坏结构状态一致性。

#### 13.3.5 本级与 ROB 级间接口

Rename 在分派时为每条 uop **占用 ROB 一项**（队尾分配）；ROB 项在**提交**前保持推测状态，并在提交时驱动**架构 RAT** 与**空闲列表回收**（§13.3.1、§13.3.2）。本表描述 **Rename → ROB** 的**分派写入**及 **ROB → Rename** 的**容量反馈**；**提交口**由 ROB 头指针侧逻辑完成，不在此表。

#### 13.3.6 每周期改名流程与多路相关（RAW / WAW / WAR）

**单槽改名**  
对每条 **`fromId.bits.valid`** 且本级资源足够的 uop，按程序序依次完成：
**（1）** 检查 **`!rob.full`**、**`rob.alloc.ready`** 等，取得本条的 **`robIdx`**；
**（2）** 解析源寄存器：用 `rs1Addr`/`rs2Addr` 查**推测 RAT**（及本节「槽间旁路」）得 `pRs1`/`pRs2`，读 **`x0`** 按实现固定；
**（3）** 若 `rfWen` 且 `rd≠x0`，从空闲列表取 **`pRd`**、读出 **`oldPhysRd`**，写 **推测 RAT[`rd`]←`pRd`**，并把 `oldPhysRd` 等写入对应 ROB 项；
**（4）** 向 **ROB**、**RS** 递交分派载荷（可与 §13.3.4、§13.3.5 **同拍、同序**）。**Flush** 时 **`specRAT ← archRAT`**，并配合全局逻辑处理 ROB / 前端。**任一步失败**则 **`fromId.ready=0`**，本周期后续槽可不分配或整拍重试（由实现定）。

**同一周期多路指令时的相关**  
设槽 **0** 为**较旧**（程序序靠前），槽 **1** 为**较新**。Rename **在同一周期内按 0→1→… 的顺序**处理各有效槽；每处理完一槽，**推测 RAT** 与**槽间旁路**对后续槽立即可见。

| 相关类型 | 含义（同周期、相邻发射槽）                                        | 处理方式                                                                                                                                                                                                                                                     |
| -------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **RAW**  | 较新指令的 `rs` 与较旧指令的 `rd` 为**同一架构寄存器**            | 较新槽在解析 `pRs1`/`pRs2` 时，不能只读「进入本周期前的 RAT」：若与较旧槽的 `rdAddr` 命中且较旧槽 **`rfWen` 且 rd≠x0**，应**直接使用较旧槽刚分配的 `pRd`**（**槽间旁路**）；否则再读 RAT。这样较新指令读到的是较旧指令的**新**物理定义，而非被覆盖前的别名。 |
| **WAW**  | 两槽均写**同一架构 `rd`**（且非 `x0`）                            | 两槽各从空闲列表取**不同**的 **`pRd`**；按程序序**依次**更新 **推测 RAT[`rd`]**：较旧槽先写入，较新槽后写入，**最终 RAT 中该架构寄存器指向较新槽的 `pRd`**。两条指令在 ROB 中各有独立项，提交时按序更新**架构 RAT**并回收旧物理寄存器，语义正确。            |
| **WAR**  | 较新指令写 `rd`，较旧指令的某 `rs` 与该 `rd` 为**同一架构寄存器** | 较旧槽**先**读 RAT 得到**旧** `pRs*`，较新槽**后**再更新 **RAT[`rd`]**；**不**需要在 Rename 停顿。WAR 在物理寄存器堆 + 重命名下**不构成真数据冒险**（读的是上一代的物理寄存器，写用新物理号）。                                                              |

**与 ID 的关系**  
若 ID 侧已按 §4.2 对**槽间 RAW** 做**降级单发射**（只发槽 0、槽 1 下周期再试），则同周期 RAW 仅来自「允许双发且仍存在相关」的指令对；**Rename 仍须具备槽间旁路**，以免在 ID 策略调整或 **N>2** 时出错。若 ID **不**降级、完全依赖后端，则 **Rename 的按序处理 + 旁路**为必需。

**Flush / 回滚** 与多路无关：一律先 **`specRAT ← archRAT`**，再按全局策略丢弃未提交 uop。

### 13.4 乱序新增保留站级（RS）

**RS** 承接 **Rename 分派**的 uop，在**源操作数就绪**且**EX级对应功能单元可接收**时向 **EX 发射**。每项登记 **`pRs1`/`pRs2` 的就绪位**（及可选已旁路数据）、**`robIdx`/`pRd`** 与控制域；通过监听 **完成总线（CDB）** 或等价广播，在 **`pRd` 生产完成**时将依赖该标签的槽置就绪。**Flush** 时清空或作废未提交项。实现可选用**统一发射队列**或**按 FU 分队列**；本规格以**单逻辑 RS 模块**描述端口，**N** 路分派 / 多 FU 时在实现侧展开。

#### 13.4.1 RS 内部存储与槽位分配

**表项（Entry）典型字段**  
RS 为每个**已分派、尚未发射完毕**的 uop 维护一行（或一条 CAM 记录），除 §13.4.2 入站载荷中的 **`robIdx`、控制域、立即数、`pc`、异常标签** 外，通常还包括：

| 类别       | 字段（概念）               | 说明                                                                                                                           |
| ---------- | -------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| 标识       | `valid` / `busy`           | 该槽是否占用                                                                                                                   |
| 重命名标签 | `pRs1`、`pRs2`、`pRd`      | 与 Rename 一致；`x0` 可不跟踪就绪                                                                                              |
| 就绪       | `src1Ready`、`src2Ready`   | 该源对应的 **`pRs*` 生产者是否已完成**（由 CDB 匹配置位）；立即数 / `PC` 路径恒就绪                                            |
| 旁路数据   | `op1Val`、`op2Val`（可选） | **仅**在 CDB 到达时**可选锁存**的 32 位值；用于 `useRsOperands=1` 时让 EX **不再读 PRF**（微优化）；**主路径不要求 RS 读 PRF** |
| 功能类型   | `opClass`                  | 绑定 ALU / LOAD / STORE / BR / MULDIV 等，用于分端口发射                                                                       |

Store 还可缓存 **`storeData`**（`pRs2` 对应值）；分支项可只关心 **`pRs*` 就绪**。**`oldPhysRd`** 可仅在 ROB 保存，RS 表项可不重复。

**槽位分配（入站）**  
使用**空闲队列（Free List）**管理可分配的槽位，无空闲槽时 **`fromRename.ready=0`**，与 §13.3 **`toRs.ready`** 相连。

**站内容器组织（逻辑次序）**  
使用**并行判断 + 优先选择**的方法，即**判断entry是否满足发射条件**并**优先选择年龄最老的指令发射**。

**发射与释放**  
当某槽**双源就绪**（及 FU 可用、`toEx.ready`）被发射后，该槽**立即释放**回空闲队列。**Flush** 时批量将槽位添加到空闲队列中。

#### 13.4.3 本级与 EX 级间接口

RS 向 EX **发射**；与 §13.5 **`fromRs.*`** **一一相连**。

**读 PRF 时机（与 §13.5.2 一致）**：默认 **`useRsOperands=0`**，EX 根据本拍收到的 **`pRs1`/`pRs2`** 与 **`op1Sel`/`op2Sel`**（见 §13.5.1 全量载荷）访问 **PRF**；仅当 RS 已在表项内锁存 CDB 数据并希望省一次读口时置 **`useRsOperands=1`**，此时 EX 用 `op*Bypass` 作为对应操作数。

除下表列出的发射侧字段外，**`toEx.bits` 与 §13.5.1 `fromRs.bits` 同构**（`imm`、`aluOp`、`op*Sel`、`wbSel`、`rfWen`、访存与分支域、`pc`、`csrAddr`、`excType`、`pRd`、`rdAddr` 等），表中未逐行重复。

| `toEx.ready`              | input  | 1    | EX 可接收（与 §13.5 `fromRs.ready` 相连）                                    |
| `toEx.bits.robIdx`        | output | *    | ROB 项编号                                                                   |
| `toEx.bits.pRs1`          | output | *    | EX 读 PRF 端口 1 地址（`op1Sel` 为寄存器源时有效）                           |
| `toEx.bits.pRs2`          | output | *    | EX 读 PRF 端口 2 地址（`op2Sel` 为寄存器源时有效）                           |
| `toEx.bits.op1Bypass`     | output | 32   | RS 已锁存的源 1 数据                                                         |
| `toEx.bits.op2Bypass`     | output | 32   | RS 已锁存的源 2 数据                                                         |
| `toEx.bits.useRsOperands` | output | 1    | **默认 0**：EX 按 §13.5.2 读 PRF；**1**：该侧用 `op*Bypass`，不读对应 PRF 口 |

#### 13.4.4 本级与完成总线（CDB）间接口

| `cdb[i].bits.result` | input | 32   | 第 `i` 路结果数据（用于入站即锁存型 RS 旁路） |
| `cdb[i].bits.robIdx` | input | *    | 第 `i` 路 ROB 项编号                          |

#### 13.4.5 Flush 与推测失效情况

| 端口           | 方向  | 位宽 | 说明                                                |
| -------------- | ----- | ---- | --------------------------------------------------- |
| `flush`        | input | 1    | 全局冲刷；RS 清空未发射项或标记无效                 |
| `flushRobTail` | input | *    | 可选：仅丢弃 `robIdx` 新于某边界的项（与 ROB 协同） |

### 13.5 执行级（EX）与其对外端口

指令经 **RS 发射**（§13.4）进入功能单元。EX 还负责 **ALU / 分支判定**（§5.1）、**访存地址与 Store 数据**送 **MEM**、**完成信息**送 **ROB/CDB**；**乘除**见 §5.5。方向均相对 **EX**；**N** 路功能单元时并列 **N** 套。

#### 13.5.1 本级与 RS 级间接口

RS 在源操作数就绪且功能单元空闲时向 EX **发射**；EX 用 **`fromRs.ready`** 反压。载荷与 §13.4.3 **`toEx.bits`** / §13.3.4 **`toRs.bits`** **同构**。**默认 `useRsOperands=0`**：EX根据 `pRs1`/`pRs2` 与 `op1Sel`/`op2Sel` 从 PRF 读取数据。**`useRsOperands=1`** 时，EX **以 `op*Bypass` 取代对 PRF 的依赖**。

| --------------------------- | ------ | ---- | ----------------------------------------------------- |

| `fromRs.ready`              | output | 1    | EX 可接收发射（含功能单元未占用、下游可接完成）       |
| `fromRs.bits.robIdx`        | input  | *    | ROB 项编号                                            |
| `fromRs.bits.pRs1`          | input  | *    | 源 1 物理寄存器号                                     |
| `fromRs.bits.pRs2`          | input  | *    | 源 2 物理寄存器号                                     |
| `fromRs.bits.pRd`           | input  | *    | 目的物理寄存器号（不写回时为无关）                    |
| `fromRs.bits.rdAddr`        | input  | 5    | 架构 `rd`（可选，调试）                               |
| `fromRs.bits.imm`           | input  | 32   | 立即数                                                |
| `fromRs.bits.aluOp`         | input  | 5    | ALU 操作码（§5.1）                                    |

| `fromRs.bits.memSigned`     | input  | 1    | Load 符号扩展                                         |
| `fromRs.bits.brType`        | input  | 3    | 分支类型                                              |
| `fromRs.bits.isJump`        | input  | 1    | JAL / JALR                                            |
| `fromRs.bits.pc`            | input  | 32   | 指令 PC                                               |

#### 13.5.2 本级与完成总线（CDB）间接口

**EX 级通过多路 CDB 广播完成信息**：EX/FU 在得到结果后，将 `robIdx/pRd/result/rfWen/异常/改向` 等打包发送到 `cdb[i]`；PRF 与 ROB 监听全部 CDB 路，分别完成物理寄存器写入与 ROB 完成登记。支持 `NCdb` 路并行完成广播；超过 `NCdb` 带宽的完成源由仲裁器裁决，失败者在后续拍重试。

| 端口                     | 方向   | 位宽 | 说明                                                             |
| ------------------------ | ------ | ---- | ---------------------------------------------------------------- |
| `cdb[i].valid`           | output | 1    | EX/FU 向第 `i` 路广播完成事件（`0 <= i < NCdb`）                 |
| `cdb[i].ready`           | input  | 1    | 第 `i` 路 CDB/仲裁器可接收                                       |
| `cdb[i].bits.robIdx`     | output | *    | 第 `i` 路完成项编号（ROB 关联键）                                |
| `cdb[i].bits.pRd`        | output | *    | 第 `i` 路目的物理寄存器号（`rfWen=1` 时有效）                    |
| `cdb[i].bits.result`     | output | 32   | 第 `i` 路结果数据（ALU/CSR/PC+4/Load 回数等，按 `wbSel` 语义）   |
| `cdb[i].bits.rfWen`      | output | 1    | 第 `i` 路是否写 PRF                                              |
| `cdb[i].bits.excType`    | output | 8    | 第 `i` 路异常标签（与译码异常合并策略由实现定义）                |
| `cdb[i].bits.mispred`    | output | 1    | 第 `i` 路分支误预测标志                                          |
| `cdb[i].bits.redirectPc` | output | 32   | 第 `i` 路误预测改向目标                                          |
| `cdb[i].bits.done`       | output | 1    | 第 `i` 路是否已得到最终结果（Load/乘除多拍可先发 0，最终拍发 1） |
| `prf.raddr1`             | output | *    | PRF 读端口 1 地址；组合读时与当拍 `fromRs` 同相                  |
| `prf.raddr2`             | output | *    | PRF 读端口 2 地址；同上                                          |
| `prf.rdata1`             | input  | 32   | PRF 读端口 1 数据；组合读时同周期返回（或经 IP 固定 1 拍延迟）   |
| `prf.rdata2`             | input  | 32   | PRF 读端口 2 数据；同上                                          |

一致性约束：同拍不得在不同 `cdb[i]` 上重复广播同一 `robIdx`；同拍同 `pRd` 的多源写回应由更前级互斥或仲裁器保证唯一性。

可见性约束：同拍 CDB 广播需在 PRF 写口、ROB 完成登记、RS 唤醒三侧语义一致；若实现存在多级流水，应保证三者观察到的是同一组完成事件（允许统一延后一拍，但不可彼此错拍）。

#### 13.5.4 失效与完成事件约束（Flush / Redirect）

分支误预测、异常或外部 Flush 到达后，EX 必须保证**错误路径 uop 不产生架构可见副作用**。对已入流水但尚未提交的 uop，要求如下：

- 若 uop 被判定失效（`robIdx` 新于 `flushRobTail`，或全局 `flush=1`），停止向 CDB 广播完成事件。
- 对已被 MEM 接收的 Store，请求取消/回滚策略由 MEM 与 ROB 协同实现；EX 不得重复发送同一 `robIdx` 的存储请求。
- 对多拍单元（乘除、长延迟 MEM），完成返回时需再次校验是否仍在有效窗口；失效项只能被丢弃，不得写 PRF/唤醒 RS/置 ROB done。
- `mispred/redirectPc` 与 CDB 完成事件可同拍上报，但同一 `robIdx` 仅允许一次最终 `done=1` 事件。

#### 13.6.4 Flush 与失效处理约束

- `flush=1` 或 `robIdx` 新于 `flushRobTail` 的访存项**必须失效**，停止向 CDB 广播。  
- 已发往内存系统的 Store 若不可撤销，需通过提交保护（如 Store Queue / Commit Gate）保证**乱序窗口外不可见**。  
- Load 返回晚于 Flush 时，只能用于内部统计，**不得写 PRF、不得唤醒 RS、不得置 ROB 完成**。  

#### 13.6.5 本级与 ROB 级间接口（Store 提交确认）

顺序访存方案下，Store 发射即按序执行，ROB 提交时确认 Store 对外可见性即可，无需复杂的 Store Queue 提交闸门。ROB 在提交 Store 指令时向 MEM 发出确认信号（若 Store 已在 MEM 级完成写入则此信号可省略）：

| 端口                              | 方向   | 位宽 | 说明                               |
| --------------------------------- | ------ | ---- | ---------------------------------- |
| `fromRob.storeCommit.valid`       | input  | 1    | ROB 确认该 Store 可对外可见        |
| `fromRob.storeCommit.ready`       | output | 1    | MEM 可接收确认                     |
| `fromRob.storeCommit.bits.robIdx` | input  | *    | 被允许对外可见的 Store 对应 ROB 项 |

#### 13.6.6 乱序核访存策略：顺序访存（In-Order Memory Access）

为降低实现复杂度，乱序核**采用顺序访存方案**：访存指令（Load/Store）在 RS 中**不允许乱序调度**，必须严格按程序序（`robIdx` 从小到大）依次发射到 MEM 级。具体规则如下：

- RS 对 `memRen=1` 或 `memWen=1` 的 uop，**仅当其为当前所有未发射访存指令中 `robIdx` 最小者**（即程序序最老）时，才允许发射。
- 非访存指令（纯 ALU、分支等）仍可乱序调度，不受上述约束。
- D-Cache Miss 时，MEM 级 Stall，RS 停止接受新的访存发射，直至 Miss 解决。

此方案天然避免了 Load 违例、Store-to-Load Forwarding 及 Load/Store Queue 等复杂机制，**不需要实现 LSQ（Load/Store Queue）**。


### 13.7 乱序新增重排序缓冲区级（ROB）与其对外接口

#### 13.7.1 ROB 存储结构

**ROB 采用环形队列维护尚未提交的 uop 顺序**，至少包含如下状态与载荷字段：

| 字段组                          | 必选             | 说明                                                               |
| ------------------------------- | ---------------- | ------------------------------------------------------------------ |
| `valid`                         | 是               | 该表项是否已分配。                                                 |
| `done`                          | 是               | 执行结果是否最终可用（由 CDB 完成事件置位）。                      |
| `killed`                        | 否               | 该项是否已被 Flush/回滚作废。                                      |
| `robIdx`                        | 是               | 全局唯一（窗口内）编号或由头尾指针可还原的索引。                   |
| `rdAddr/pRd/oldPhysRd/rfWen`    | 是               | 提交时用于更新 `archRAT` 与回收旧物理寄存器。                      |
| `result`                        | 否               | 若选择在 ROB 暂存结果，则由完成事件写入；也可只存 PRF 标签不存值。 |
| `excType/mispred/redirectPc`    | 是               | 异常与改向信息（精确异常与恢复路径）。                             |
| `isLoad/isStore/storeCommitted` | 前二者是，后者否 | 访存类提交控制所需元信息。                                         |

**指针与容量约束**：

- `head` 指向最老未提交项，`tail` 指向下一可分配槽；空满判定可用计数或保留一空槽策略。
- 分配（Rename）与完成（CDB）可乱序发生，但提交**严格按 `head` 程序序**进行。
- 仅当队首项满足 `valid=1 && done=1 && killed=0` 且无更老异常阻塞时允许提交。
- 为支持精确异常，队首若为异常项，应先发起 Flush/Redirect，再提交该异常（或按实现定义终止），**禁止更年轻项先提交**。

**推荐状态转移**：`invalid -> allocated -> done -> committed -> invalid`，Flush 时可从 `allocated/done` 直接进入 `invalid`（或 `killed` 后回收）。

**何时发生状态转移（按拍语义）**：

| 状态转移                             | 触发时机（本拍）           | 关键条件                                                                                                     | 动作                                                                        |
| ------------------------------------ | -------------------------- | ------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------- |
| `invalid -> allocated`               | Rename 到 ROB 分配握手成立 | `dispatch.valid && dispatch.ready`                                                                           | `tail` 指向项写入元数据并置 `valid=1, done=0, killed=0`；拍末 `tail` 前移。 |
| `allocated -> done`                  | 任一路 CDB 完成握手成立    | `cdb[i].valid && cdb[i].ready` 且 `cdb[i].bits.robIdx` 命中且 `cdb[i].bits.done=1`                           | 置 `done=1`，并合并 `result/excType/mispred/redirectPc` 等完成信息。        |
| `allocated -> invalid`（Flush 直删） | 全局冲刷生效               | 项被判定为年轻项（如 `robIdx > flushRobTail`）                                                               | 直接清空为 `invalid`，不等待完成事件。                                      |
| `allocated -> killed`（可选实现）    | 冲刷后延迟回收策略启用     | 实现选择保留墓碑态                                                                                           | 先置 `killed=1`，后续统一回收阶段转 `invalid`。                             |
| `done -> committed`                  | 队首提交链路握手可完成     | 该项位于 `head`，且提交相关握手（如 `toArchRat.commit.valid&&ready`、`toFreeList.release.valid&&ready`）成立 | 视为提交成功并触发架构状态更新。                                            |
| `committed -> invalid`               | 提交成功同拍或下一拍       | 无额外条件                                                                                                   | 清空表项（`valid=0`），释放槽位，`head` 前移到下一项。                      |
| `done -> invalid`（异常/误预测回滚） | 提交前命中错误路径冲刷     | 该项虽 `done=1` 但被判定失效                                                                                 | 按 Flush 规则直接作废回收，不产生架构可见提交。                             |

特别地，同一表项单拍内若同时出现“完成事件”和“Flush 失效”，**Flush 总是优先**；即该项不得进入可提交态。

#### 13.7.2 本级与完成总线（CDB）间接口

ROB 监听 `cdb[i]`（`0 <= i < NCdb`）并按 `robIdx` 更新项状态。接口采用 **Ready-Valid**：仅当 `valid && ready` 成立时消费该路完成事件。

| 端口                  | 方向   | 位宽 | 说明                                    |
| --------------------- | ------ | ---- | --------------------------------------- |
| `cdb[i].valid`        | input  | 1    | 第 `i` 路完成事件有效                   |
| `cdb[i].ready`        | output | 1    | ROB 可接收第 `i` 路完成事件             |
| `cdb[i].bits.robIdx`  | input  | *    | 完成项索引，命中后置对应 ROB 项 `done`  |
| `cdb[i].bits.pRd`     | input  | *    | 写回物理寄存器号（用于一致性校验/调试） |
| `cdb[i].bits.result`  | input  | 32   | 结果数据（若 ROB 选择存值）             |
| `cdb[i].bits.rfWen`   | input  | 1    | 是否写 PRF                              |
| `cdb[i].bits.excType` | input  | 8    | 新异常标签（与既有异常合并）            |
| `cdb[i].bits.done`    | input  | 1    | 最终完成标志                            |

**一致性约束**：同拍不得出现两个 `cdb[i]` 指向同一 `robIdx`；同一 ROB 项仅允许一次最终 `done=1` 事件。

#### 13.7.3 本级与架构 RAT 间接口

ROB 在提交写寄存器指令时更新 `archRAT`，使架构可见映射前移到该提交点：

| 端口                           | 方向   | 位宽 | 说明                                   |
| ------------------------------ | ------ | ---- | -------------------------------------- |
| `toArchRat.commit.valid`       | output | 1    | ROB 队首提交且需更新映射               |
| `toArchRat.commit.ready`       | input  | 1    | ArchRAT 可接收提交更新                 |
| `toArchRat.commit.bits.rdAddr` | output | 5    | 架构目标寄存器（`rd != x0`）           |
| `toArchRat.commit.bits.pRd`    | output | *    | 新提交生效的物理寄存器号               |
| `toArchRat.commit.bits.robIdx` | output | *    | 可选：提交来源 ROB 项编号（调试/追踪） |

#### 13.7.4 本级与空闲队列（Free List）间接口

ROB 在提交后回收被覆盖的旧物理寄存器（`oldPhysRd`）：

| 端口                          | 方向   | 位宽 | 说明                             |
| ----------------------------- | ------ | ---- | -------------------------------- |
| `toFreeList.release.valid`    | output | 1    | 本拍有旧物理寄存器可回收         |
| `toFreeList.release.ready`    | input  | 1    | 空闲队列可接收                   |
| `toFreeList.release.bits.pRd` | output | *    | 待回收物理寄存器号（来自提交项） |

#### 13.7.5 Flush / Redirect 协同

当收到误预测改向或异常冲刷时，ROB 输出全局恢复边界并丢弃年轻项：

| 端口                          | 方向   | 位宽 | 说明                                    |
| ----------------------------- | ------ | ---- | --------------------------------------- |
| `toBackendFlush.valid`        | output | 1    | 冲刷事件有效                            |
| `toBackendFlush.ready`        | input  | 1    | 后端各级已接收冲刷事件                  |
| `toBackendFlush.bits.robTail` | output | *    | 保留边界（仅 `robIdx <= robTail` 有效） |
| `toBackendFlush.bits.reason`  | output | 2    | 原因：误预测/异常/外部请求（编码自定）  |
| `toFrontendRedirect.valid`    | output | 1    | 通知前端改向                            |
| `toFrontendRedirect.ready`    | input  | 1    | 前端可接收改向                          |
| `toFrontendRedirect.bits.pc`  | output | 32   | 改向目标 PC                             |

### 13.8 与本文档其余章节的衔接

ALU 真值表与操作数编码见 **§5.1、§5.2**；D-Cache / 总线见 **§6、§2、§10**；CSR 写回语义见 **§9**；顺序核 **§5.3** 旁路优先级在乱序中由 **PRF + CDB/唤醒（§13.4.4）** 替代。§13.4（RS）、§13.5（EX）端口为**功能边界**，流水级数、多 FU 仲裁与 CDB 路数由突击组自定；**EX 发出的 CDB 完成事件**须在 ROB 与 RS 监听侧保持一致，避免唤醒或记账丢失。

### 13.9 ROB 提交宽度规范

**ROB 提交宽度（commit width）固定为 N=2**，即每周期最多提交 2 条已完成的程序序连续 uop。提交从 `head` 指针起，依次检查：`valid=1 && done=1 && killed=0` 且无更老异常；满足则提交并推进 `head`，最多推进 2 格。

- 若 `head` 项有异常（`excType ≠ 0`），本周期只允许处理该异常项（提交宽度退化为 1），不允许同周期继续提交更年轻项。
- 提交宽度影响 IPC 上限（理论峰值 IPC = commit width / CPI_baseline）；实现者不得将 commit width 缩减为 1（除非参数化关闭双宽度）。
- `minstret` 计数以实际本周期提交条数为准（0、1 或 2）。

### 13.10 RS 调度策略规范

保留站（RS）采用 **Oldest-First（程序序最老优先）** 调度策略：当多个已就绪（所有源操作数均可用）的 uop 同时竞争同一执行单元时，选择 `robIdx` 最小（即最早分配、程序序最老）的 uop 发射。

- 理由：Oldest-First 可减少 ROB head 阻塞概率，有助于提升实际提交率和 IPC。
- 若同一周期有多个执行单元可并发发射，每个 FU 独立选择其队列中的最老就绪项。
- 实现侧可采用优先编码器（priority encoder）基于 `robIdx` 实现，也可采用年龄矩阵（age matrix）。

### 13.11 Rename 资源耗尽与背压规范

当空闲列表（Free List）中可用物理寄存器不足以满足本周期 Rename 需求（需要 `N` 枚新寄存器而剩余 < `N`）时：

- Rename 级必须**停止分配**，拉低 `toRename.ready`（即 `fromId.ready = 0`），向上游 ID 和 IF 传递背压，令 ID/IF 级 **stall**。
- **Stall 必须向前端传播**：`stallIF = 1`，`stallID = 1`，ID/EX 流水线寄存器冻结，直至 Free List 回收到足够资源（ROB 提交时释放旧物理寄存器）。
- 同理，若 ROB 已满（`tail + N > head + ROB_SIZE`），亦触发相同的 Rename stall 背压。
- 两种资源耗尽（Free List 和 ROB 满）可共享同一 stall 信号，取 OR 后传递给 ID 和 IF。

### 13.12 乱序核异常处理

本课设**不要求**乱序核实现精确异常（Precise Exception）机制。ROB 中的 `excType` 字段可保留用于调试，但不需要触发陷入、CSR 写入或 PC 跳转至 mtvec 等流程。误预测引发的 Rollback 仍须正确实现（见 §13.3.1）。
